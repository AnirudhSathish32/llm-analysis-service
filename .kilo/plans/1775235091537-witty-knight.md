# Codebase Recon Report: LLM Analysis Service

> Generated: 2026-04-03
> Scope: All Python files, config, Docker, CI/CD, tests

---

## 1. SECURITY ISSUES

### CRITICAL

**S1. Hardcoded DB credentials in config defaults**
- `app/core/config.py:8-11` — `database_url` defaults to `"postgresql+asyncpg://llm_user:llm_password@db:5432/llm_service"`. If `DATABASE_URL` env var is missing, the fallback contains a username/password in the codebase.

**S2. No authentication/authorization on any endpoint**
- All routes (`analysis.py`, `documents.py`, `metrics.py`) are publicly accessible. No API key validation, JWT, or OAuth. The README shows the service is live at a public IP with no auth.

**S3. File upload has no size limit**
- `app/api/routes/documents.py:21-51` — No `max_file_size` check. An attacker could upload a multi-gigabyte file, exhausting disk space.

**S4. File upload path traversal risk**
- `app/api/routes/documents.py:48` — `file.filename` is used directly in path construction. Could contain path separators on some platforms. Should sanitize.

**S5. Internal error details leaked to clients**
- `app/api/routes/documents.py:85` — `detail=str(e)` returns the raw exception message to the client, potentially leaking internal paths, SQL, or stack traces.

**S6. Public IP exposed in README**
- `README.md:9` — `http://18.222.255.28:8000/docs` exposes a live production IP in a public repository.

### HIGH

**S7. `PINECONE_API_KEY` missing from `.env`**
- `.env` — `PINECONE_API_KEY` is never defined, but `rag_pipeline.py:48` reads it via `os.environ["PINECONE_API_KEY"]`. This will crash at runtime.

**S8. AWS account ID and Secrets ARNs hardcoded in task-definition.json**
- `task-definition.json` — AWS account `593489476293` and full Secrets Manager ARNs are hardcoded. If repo is public, it reveals infrastructure details.

## 2. INEFFICIENCIES / PERFORMANCE

**P1. Gemini SDK call is synchronous, blocking the event loop**
- `app/services/llm_router.py:96-118` — `_call_gemini` uses synchronous `model.generate_content(prompt)`. Blocks the entire asyncio event loop during the HTTP call. Should use `run_in_executor` or the async Gemini SDK.

**P2. LLM clients created on every call**
- `app/services/llm_router.py:73` — `anthropic.AsyncAnthropic(api_key=...)` instantiated inside `_call_anthropic` on every call.
- `app/services/llm_router.py:100` — `genai.configure()` and `genai.GenerativeModel()` on every Gemini call.
- Should be module-level singletons.

**P3. Pinecone client created on every request**
- `app/services/rag_pipeline.py:47-49` — `_get_index()` creates a new `Pinecone()` instance on every call. Should be a singleton.

**P4. Google API key configured on every embedding call**
- `app/services/rag_pipeline.py:25, 35` — `genai.configure(api_key=...)` called inside both `_embed_texts` and `_embed_query_text`. Should be called once at module init.

**P5. No connection pooling configuration**
- `app/db/session.py:4` — `create_async_engine` has no `pool_size`, `max_overflow`, or `pool_timeout` settings. Defaults may be suboptimal for production.

**P6. Cache TTL default is 24 hours**
- `app/core/config.py:16` — `cache_ttl_seconds: int = 86400`. Extremely aggressive for LLM responses. If env var is missing, stale cached results persist for a full day.

**P7. Orphaned files on ingestion failure**
- `app/api/routes/documents.py:46-62` — File is saved to disk, then DB record created, then ingestion runs. If ingestion fails, the orphaned file on disk is never cleaned up.

**P8. No Redis connection lifecycle management**
- `app/cache/redis.py:4` — `redis_client` created at module import with no explicit connection, no `ping()` health check, no close on shutdown.

## 4. ARCHITECTURE ISSUES

**AR1. `create_all_tables` used instead of Alembic in lifespan**
- `app/main.py:9, 16` — App uses `create_all_tables()` at startup, not Alembic migrations. Alembic is set up (`migrations/`, `alembic.ini`) but never invoked. They can drift out of sync.

**AR2. No graceful shutdown**
- `app/main.py:25` — `yield` has no shutdown logic. Redis connection is never closed. No cleanup of temp files.

**AR3. Rate limiter has no Redis error handling**
- `app/core/rate_limit.py:7-15` — If Redis is down, `redis_client.incr(key)` raises an exception that becomes a 500. Should fail open (allow) or fail closed (deny) explicitly.

**AR4. Rate limit is not configurable**
- `app/core/rate_limit.py:4-5` — `RATE_LIMIT = 20` and `WINDOW_SECONDS = 60` are hardcoded module-level constants, not read from `settings`.

**AR5. No request correlation/tracing IDs**
- No middleware adds a `X-Request-ID` header. When something fails, there is no way to correlate logs across the request lifecycle.

**AR6. Metrics endpoints have no pagination or time filter**
- `app/api/routes/metrics.py:11-31` — `usage()` aggregates across ALL rows in the database. As the table grows, this query will become slower. No time range filter.

**AR7. `AnalysisService` is not dependency-injected**
- `app/api/routes/analysis.py:10` — `service = AnalysisService()` instantiated at module level. Hard to mock in tests, prevents per-request configuration.

**AR8. `llm_router.py` mixes concerns**
- `app/services/llm_router.py` — Contains cost calculation, prompt building, Anthropic client code, Gemini client code, and router logic. Should be separated into `prompts.py`, `providers/`, `router.py`.

**AR9. Dockerfile copies `.env` into the image**
- `Dockerfile:19` — `COPY . .` copies everything including `.env` into the container. Should be excluded via `.dockerignore`.

**AR10. No `.dockerignore` file**
- `venv/`, `__pycache__/`, `.git/`, `.kilo/`, and other unnecessary files are copied into the Docker image, bloating it.

**AR11. Docker `--reload` in development compose**
- `docker-compose.yaml:48` — `--reload` flag is for development only. `task-definition.json` does not use `--reload`, so this is inconsistent (acceptable for dev, but worth noting).

**AR12. Dockerfile USER directive placement**
- `Dockerfile:20-21` — `adduser` and `USER appuser` come after `COPY . .`. Uploaded files in `/tmp/uploads` may not be writable by `appuser`.

## 7. DEVEX ISSUES

**DX1. `llm_router.py` uses `os.environ` directly instead of `settings`**
- `app/services/llm_router.py:73, 100` — Uses `os.environ["ANTHROPIC_API_KEY"]` and `os.environ["GEMINI_API_KEY"]` instead of `settings`. Bypasses pydantic-settings validation, raises `KeyError`.

**DX2. `rag_pipeline.py` uses `os.environ` and `os.getenv` mixed**
- `app/services/rag_pipeline.py:14-17` — `os.getenv()` with defaults
- `app/services/rag_pipeline.py:25, 35, 48` — `os.environ[]` (no default, raises KeyError)
- Should consistently use `settings` from `app.core.config`.

**DX3. `PINECONE_API_KEY`, `ANTHROPIC_API_KEY`, `GEMINI_API_KEY` missing from `Settings`**
- `app/core/config.py` — None of these are defined in `Settings` but are required by `llm_router.py` and `rag_pipeline.py`. Should be `SecretStr` fields.

**DX4. Flake8 ignores are excessive**
- `.github/workflows/ci.yml:54` — `--ignore=E501,W503,E302,E305,E122,E241,W292,W391,W293,F401,F821`
- `F401` (unused imports) and `F821` (undefined name) are critical checks that should never be ignored.

**DX5. No `pyproject.toml` or `setup.cfg`**
- Project configuration is scattered: `requirements.txt` for deps, `alembic.ini` for migrations, no central config for pytest, flake8, mypy, black.

**DX6. No lock file**
- `requirements.txt` has pinned versions but no `requirements.lock` or `poetry.lock`. Reproducible builds not guaranteed.

**DX7. `pytest-asyncio` mode not configured**
- No `pytest.ini`, `pyproject.toml`, or `conftest.py` to set `asyncio_mode = "auto"`. Each test must manually use `@pytest.mark.asyncio`.

**DX8. Missing type hints**
- `app/db/session.py:7` — `get_session()` has no return type. Should be `-> AsyncGenerator[AsyncSession, None]`.
- `app/core/rate_limit.py:7` — `rate_limiter()` has no return type. Should be `-> None`.
